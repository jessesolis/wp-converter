# WORDPRESS-OUTPUT.md — WordPress Output Structure and Conventions

Read this before generating any WordPress output files.

---

## Output Package Structure

```
export.zip
├── theme/
│   └── scorpion-converted/          ← WordPress child theme
│       ├── style.css                ← Theme header (required by WP)
│       ├── functions.php            ← Registers all assets, enqueues per-page
│       ├── index.php                ← Minimal fallback template
│       ├── css/
│       │   ├── bundle-1.css         ← Extracted Scorpion stylesheets
│       │   ├── bundle-2.css
│       │   ├── inline-home.css      ← Per-page inline <style> blocks
│       │   ├── inline-about.css
│       │   └── ...
│       ├── js/
│       │   ├── utility-1.js         ← Extracted Scorpion JS utilities
│       │   └── ...
│       └── templates/
│           ├── page-home.php        ← Per-page static HTML templates
│           ├── page-about.php
│           └── ...
├── media/
│   ├── image-1.jpg                  ← All downloaded media assets
│   └── ...
├── import.xml                       ← WXR file for WordPress importer
└── MIGRATION-CHECKLIST.md           ← Post-import steps for the user
```

## WordPress Child Theme

### style.css (required theme header)
```css
/*
Theme Name: Scorpion Converted
Description: Auto-generated by Scorpion CMS → WordPress Conversion Tool
Version: 1.0
*/
```

### functions.php — stylesheet and JS enqueuing
Every CSS/JS file is **registered** globally (`wp_register_style` / `wp_register_script`). Each page then **enqueues only the assets the original Scorpion page actually loaded**, in their original document order, looked up by the page's template slug. Per-page inline `<style>` blocks are written to `inline-<templateSlug>.css` and appended after the page's bundles so they cascade correctly.

```php
<?php
function scorpion_converted_register_assets() {
    wp_register_style('scorpion-style-1', get_stylesheet_directory_uri() . '/css/bundle-1.css', [], null);
    wp_register_style('scorpion-style-2', get_stylesheet_directory_uri() . '/css/bundle-2.css', [], null);
    wp_register_style('scorpion-style-3', get_stylesheet_directory_uri() . '/css/inline-home.css', [], null);
    // ... one wp_register_* per CSS / JS file shipped in the theme

    wp_register_script('scorpion-script-1', get_stylesheet_directory_uri() . '/js/utility-1.js', [], null, true);
    // ...
}
add_action('wp_enqueue_scripts', 'scorpion_converted_register_assets', 5);

// Template slug → asset handles the original Scorpion page loaded, in order.
function scorpion_converted_page_assets_map() {
    return array(
        'home' => array(
            'css' => array('scorpion-style-1', 'scorpion-style-2', 'scorpion-style-3'),
            'js'  => array('scorpion-script-1'),
        ),
        'about' => array(
            'css' => array('scorpion-style-1', 'scorpion-style-4'),
            'js'  => array('scorpion-script-1'),
        ),
        // ...
    );
}

function scorpion_converted_enqueue_page_assets() {
    if (!is_page()) return;
    $tpl = get_page_template_slug();
    if (!preg_match('#templates/page-(.+)\.php$#', $tpl, $m)) return;
    $slug = $m[1];
    $map = scorpion_converted_page_assets_map();
    if (!isset($map[$slug])) return;
    foreach ($map[$slug]['css'] as $handle) wp_enqueue_style($handle);
    foreach ($map[$slug]['js']  as $handle) wp_enqueue_script($handle);
}
add_action('wp_enqueue_scripts', 'scorpion_converted_enqueue_page_assets', 10);
```

> **Important:** JS files are registered with `$in_footer = true` (last argument). This mirrors how Scorpion loads its utilities and prevents render-blocking. Pages that aren't WordPress pages (404, archive, etc.) skip the enqueue and render with no Scorpion assets — that's intentional, those routes aren't part of the original Scorpion site.

### Page templates
Each converted page gets its own PHP template file. The static HTML from Scorpion is used as the template body, with `.cnt-stl` regions replaced by Classic block output.

```php
<?php
/* Template Name: Home Page */
get_header();
?>

<!-- Static HTML structure from Scorpion, verbatim -->
<section class="mainstage">
  <div class="mainstage-inner">
    <!-- WP Classic Block content injected here by WordPress -->
    <?php the_content(); ?>
  </div>
</section>

<!-- ... rest of static HTML ... -->

<?php get_footer(); ?>
```

## Gutenberg Classic Blocks

Each `.cnt-stl` region becomes a **Classic block** in the WordPress page content.

### Block format in WXR XML
```xml
<!-- wp:freeform -->
<div class="cnt-stl">
  <!-- inner HTML extracted from Scorpion, verbatim -->
  <h2>Our Services</h2>
  <p>We provide legal representation...</p>
</div>
<!-- /wp:freeform -->
```

> Use `wp:freeform` — this is the WXR representation of the Classic block. It preserves arbitrary HTML without Gutenberg trying to interpret or transform it.

### Ordering
Multiple `.cnt-stl` blocks on a single page are preserved in **DOM order** — the sequence they appeared in the original Scorpion page.

## WXR XML (WordPress Import File)

The WXR file is consumed by WordPress's built-in importer (`Tools → Import → WordPress`).

### Required WXR structure
```xml
<?xml version="1.0" encoding="UTF-8"?>
<rss version="2.0"
  xmlns:excerpt="http://wordpress.org/export/1.2/excerpt/"
  xmlns:content="http://purl.org/rss/1.0/modules/content/"
  xmlns:wfw="http://wellformedweb.org/CommentAPI/"
  xmlns:dc="http://purl.org/dc/elements/1.1/"
  xmlns:wp="http://wordpress.org/export/1.2/">
  <channel>
    <title>Site Title</title>
    <link>https://example.com</link>
    <wp:wxr_version>1.2</wp:wxr_version>
    <wp:base_site_url>https://example.com</wp:base_site_url>
    <wp:base_blog_url>https://example.com</wp:base_blog_url>

    <!-- Authors -->
    <wp:author>...</wp:author>

    <!-- Navigation menus -->
    <wp:term>...</wp:term>

    <!-- Pages, posts, nav items as <item> elements -->
    <item>...</item>
  </channel>
</rss>
```

### Page item structure
```xml
<item>
  <title>About Us</title>
  <link>https://example.com/about</link>
  <pubDate>Mon, 01 Jan 2024 00:00:00 +0000</pubDate>
  <dc:creator>admin</dc:creator>
  <content:encoded><![CDATA[
    <!-- wp:freeform -->
    <div class="cnt-stl">...</div>
    <!-- /wp:freeform -->
  ]]></content:encoded>
  <wp:post_type>page</wp:post_type>
  <wp:status>publish</wp:status>
  <wp:post_name>about</wp:post_name>
  <wp:page_template>templates/page-about.php</wp:page_template>
  <!-- SEO metadata as postmeta -->
  <wp:postmeta>
    <wp:meta_key>_yoast_wpseo_title</wp:meta_key>
    <wp:meta_value><![CDATA[About Us | Site Name]]></wp:meta_value>
  </wp:postmeta>
  <wp:postmeta>
    <wp:meta_key>_yoast_wpseo_metadesc</wp:meta_key>
    <wp:meta_value><![CDATA[Meta description here]]></wp:meta_value>
  </wp:postmeta>
</item>
```

### Navigation menus
```xml
<!-- Register menu -->
<wp:term>
  <wp:term_id>1</wp:term_id>
  <wp:term_taxonomy>nav_menu</wp:term_taxonomy>
  <wp:term_slug>primary-menu</wp:term_slug>
  <wp:term_name><![CDATA[Primary Menu]]></wp:term_name>
</wp:term>

<!-- Menu items as nav_menu_item post type -->
<item>
  <wp:post_type>nav_menu_item</wp:post_type>
  <wp:status>publish</wp:status>
  <category domain="nav_menu" nicename="primary-menu"><![CDATA[Primary Menu]]></category>
  <wp:postmeta>
    <wp:meta_key>_menu_item_type</wp:meta_key>
    <wp:meta_value><![CDATA[custom]]></wp:meta_value>
  </wp:postmeta>
  <wp:postmeta>
    <wp:meta_key>_menu_item_url</wp:meta_key>
    <wp:meta_value><![CDATA[/about]]></wp:meta_value>
  </wp:postmeta>
</item>
```

## SEO Metadata

SEO metadata is stored as post meta. The tool writes meta keys compatible with **Yoast SEO** (most common WP SEO plugin). If a different SEO plugin is needed, only the meta key names change.

| Scorpion field | Yoast meta key |
|---|---|
| Page `<title>` | `_yoast_wpseo_title` |
| Meta description | `_yoast_wpseo_metadesc` |
| Canonical URL | `_yoast_wpseo_canonical` |
| OG title | `_yoast_wpseo_opengraph-title` |
| OG description | `_yoast_wpseo_opengraph-description` |

## Media Asset Re-referencing

After media is downloaded to `/tmp/scorpion-conversions/{job_id}/media/`, all references to those assets in page HTML and extracted `.cnt-stl` content must be updated to point to the WordPress uploads path:

- Original: `https://scorpion-site.com/assets/images/hero.jpg`
- WordPress: `/wp-content/uploads/scorpion-migration/hero.jpg`

This replacement must happen in:
- All page template HTML
- All Classic block inner HTML
- Any CSS `url()` references within extracted stylesheets that point to site-hosted assets (e.g. background images)

## Migration Checklist (MIGRATION-CHECKLIST.md)

Included in every export `.zip`. Covers post-import manual steps:

1. Install and activate the child theme
2. Run WordPress importer with `import.xml`
3. Assign the primary menu to the correct menu location
4. Install Yoast SEO (or preferred SEO plugin) and verify metadata imported correctly
5. Upload media folder contents to `/wp-content/uploads/scorpion-migration/`
6. Review all forms — reconnect notification email addresses
7. Test all interactive components (sliders, accordions, tabs)
8. Run a visual comparison against the original Scorpion site
9. Check mobile and tablet layouts
10. Verify Google Fonts are loading correctly
